## Key decisions made (carried forward)

- **Python version:** Switched from 3.11 → **3.12** (only version installed on user's machine; FastAPI fully supports it; `homelab-context.md` §8 updated accordingly)
- **Server:** Uvicorn with `[standard]` extras (auto-reload, better logging, websocket support)
- **FastAPI install:** `fastapi[standard]` for the recommended optional deps
- **Dependency pinning style:** Compatible-release (`~=`) — accepts patch updates, blocks minor/major version jumps
- **Hardcoded data range:** 10.10.14.x (matches user's actual VM range from `homelab-context.md` §3)
- **Endpoint naming:** RESTful nouns (`/ips`, not `/getIps`) — convention for future expansion
- **Response shape:** Wrapped lists in objects with `count` + data — leaves room for pagination/metadata later
- **Encoding standard:** All source files saved as UTF-8 (no BOM); confirmed via `Select-String` checks

## Issues encountered & resolved

1. **Broken venv on first creation** — `python -m venv .venv` produced a folder containing only `python.exe`, missing pip and activation scripts. Fixed by recreating with `python -m venv .venv --upgrade-deps`. Likely a one-off issue with the Python 3.12 install.

2. **Python version mismatch with plan** — context doc said 3.11, but only 3.12 was installed on the laptop. Decided to update the plan to 3.12 (zero practical impact for this stack; FastAPI, SQLAlchemy, Alembic, icmplib all support 3.12 fully).

3. **UTF-8 encoding artifacts** — em-dashes in code comments rendered as `â€"` because Notepad defaulted to ANSI encoding. Fixed by replacing with plain hyphens and saving via VS Code with explicit UTF-8 (no BOM) encoding.

**Mental shift for Phase 2:** the API stays the same shape; the data source moves from a Python list to a real database. Same JSON out, different code path inside. This is the principle of an *API contract* — clients don't care how data is stored, only how it's exposed.
